## Top offenders (ranked by impact-per-effort)

### 1. Serial Anthropic calls in cascade waves

**Evidence**:
- `app/api/chat/route.ts:558-590`. `backgroundFireScout` loops through
  `cats` and `await`s `scoutShortlist` **sequentially**. Wave 2 is
  `["Caterer", "Florist", "Officiant"]` (3 calls). Wave 4 is
  `["Band", "DJ", "Hair & Makeup", "Rentals"]` (4 calls).
- `lib/cascade.ts:155-194`. `handleLockBrief` does the same serial loop
  for `["Venue", "Photographer"]`.

**Hypothesis**: each `scoutShortlist` makes an Anthropic API call that
takes 5-15s. Three serial calls = 15-45s. They have no shared state
beyond reads, so they can fire in parallel.

**Fix**: convert the `for...of` await loop to `Promise.all(cats.map(...))`
in both files. Each shortlist + addVendors + appendApproval still
serializes its own writes (Anthropic call → DB write → approval card),
but across categories they fan out.

**Expected outcome**: cascade completion time reduced from N×latency to
~max(latency) where N is categories per wave. Real-world: 30s → 12s on
a 3-category wave with the live Anthropic path; offline-fixture path is
already ~10ms so impact there is negligible.

**Rollback**: revert the two diffs and restore the `for...of await`
loops.

### 2. `googleapis` (195 MB) imports the entire Google APIs SDK

**Evidence**:
- `node_modules/googleapis` weighs **195 MB**.
- Only consumer is `lib/gmail/client.ts:10`. `import { google, gmail_v1 } from "googleapis";`
- Corsia only uses Gmail + OAuth2. that's two of the ~300 services the
  SDK ships.

**Hypothesis**: serverless cold-start time is proportional to function
package size. `googleapis` adds ~180 MB of code paths Corsia never
exercises. On Vercel, that's a measurable cold-start hit. (NOT MEASURED
in sandbox.)

**Fix candidate**: swap to `@googleapis/gmail` (scoped Gmail-only
package, ~5-15 MB) + `google-auth-library` (already a transitive dep,
~776 KB). API surface differs slightly. `google.gmail({...})` becomes
`gmail({ version: "v1", auth })`. The OAuth2 client construction stays.

**Status**: **DEFERRED**. The swap is invasive enough (touches every
gmail call) and the metric (cold-start) isn't measurable in this
sandbox, so I won't ship it without evidence the change moves the
number on real infrastructure. Filed in the result doc as a follow-up.

### 3. Full-state JSON write on every mutation

**Evidence**:
- `lib/store.ts:113-121`. `writeState` does
  `JSON.stringify(next, null, 2)` + `fs.writeFile` of the entire state
  on every mutation, even one-field changes.
- `lib/store.ts:125-127`. `mutate` does `structuredClone(cur)` per
  mutation; the entire state is deep-cloned before the reducer runs.

**Hypothesis**: at current state size (demo state ~70 KB JSON), this
is sub-millisecond. At a larger state with hundreds of guests, hundreds
of approvals, and a populated ledger, this would scale linearly.

**Status**: **DEFERRED**. Not on the slow path today; the file ops are
~1 ms at current state size. Worth re-measuring once a real wedding
project is loaded.

## Things investigated but not fixed

- **Dashboard fetch waterfall**: not a waterfall. Today.tsx fires one
  `/api/watcher` and `StateProvider` fires one `/api/state`. Both run
  in parallel via React mount.
- **N+1 query patterns**: `lib/store.ts` is a single-file JSON store
  with one read + one write per request. No N+1.
- **Bundle composition**: build timed out in sandbox; cannot quote
  route bundle sizes. Re-measure on a Mac.
