# Performance Audit. Results. 2026-05-10

## Sandbox-honest preface

Most of the audit's ideal measurements (LCP, TTI, total blocking time,
React profiler, Chrome DevTools traces) cannot be captured in the
sandbox running this pass. The fixes below were chosen on the basis of
**static analysis**. code patterns visible in the source. and the
expected-latency math for the live Anthropic API path. The follow-up
session on a Mac with Chrome should run the full Phase 1 + Phase 5
measurement to confirm real-world wall-clock numbers.

Honest caveat: the offline scout path (used when no `ANTHROPIC_API_KEY`
is set) is already ~10 ms per category, so on the demo path the fixes
below are a no-op. The improvement applies in **live mode** with a real
Anthropic key. i.e. real users.

## What was fixed

### 1. `app/api/chat/route.ts`. `backgroundFireScout` parallelized

Converted the serial `for (const cat of cats) await scoutShortlist(...)`
to `await Promise.all(cats.map(async (cat) => { ... }))`. Per-category
DB writes (addVendors → appendApproval) still serialize within a single
category to preserve write ordering, but across categories the
Anthropic calls fire in parallel. Wave 2 (Caterer + Florist + Officiant)
and Wave 4 (Band + DJ + Hair & Makeup + Rentals) both benefit.

### 2. `lib/cascade.ts`. `handleLockBrief` parallelized

Same pattern. Lock cascade fires Venue + Photographer in parallel
instead of serially. Refactored to also pull the brief reference out
of the closure so each parallel branch reads from a stable local
const, avoiding repeated `state.brief` accesses inside the map.

## What was tried but reverted

Nothing. both fixes compiled cleanly on the first pass and didn't
change observable product behavior (the same writes happen in the same
order within each category branch).

## What was deferred

### `googleapis` (195 MB) → scoped `@googleapis/gmail`

Filed but not shipped. The `googleapis` package adds ~180 MB of unused
service surface; Corsia only uses Gmail and OAuth2. Swapping to
`@googleapis/gmail` plus the existing `google-auth-library` transitive
dep would shrink `node_modules` by ~180 MB and improve serverless
cold-start time on Vercel. The change is a single-file refactor in
`lib/gmail/client.ts`, but the API surface differs slightly between
the two packages (e.g. `google.gmail({...})` becomes `gmail({...})`)
and there's no way to verify cold-start improvement from the sandbox.
Recommended for the next session, on a Mac with `next build` access.

### `lib/store.ts` full-state writes

Filed but not shipped. Every mutation does
`JSON.stringify(state, null, 2)` of the full state plus a
`structuredClone(cur)`. At current state size (~70 KB demo JSON) this
is sub-millisecond, but it scales linearly with state size. Worth
re-measuring once a real wedding has hundreds of guests and a populated
ledger.

## What was cleaned up

Nothing. `depcheck` reported zero unused production dependencies, and
ts-prune / knip weren't installed in the sandbox. No dead-code
deletions this pass.

## Honest assessment

This was a measurement-constrained pass. I could only reason from
static analysis. bundle size on disk, fetch shape in the dashboard,
serial-await loops in the chat route. The two fixes I shipped target
the most expensive code path I could see (the lock cascade's serial
Anthropic calls) and are theoretically 2-4× faster in live mode. They
are 0% faster in offline/demo mode because the offline scout returns
immediately.

What's still slow that this pass didn't touch:
- Anything client-side (LCP, TTI, render time of CommandCenter on a
  populated demo state). Need Chrome.
- Cold-start of Vercel functions if `googleapis` stays. Need Vercel.
- Bundle composition per route. Need a working `next build` analyze.

The next perf session should run on a Mac, capture real Chrome traces
of the dashboard + CommandCenter, run `@next/bundle-analyzer`, and
decide whether the `googleapis` swap is worth the refactor against
measured cold-start time.
